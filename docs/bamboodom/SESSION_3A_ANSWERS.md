## Блок А. Архитектура

### А1. Без BasePublisher ✅
Правильно. Для e2e-теста и ручного ввода FSM внутри админки — достаточно. Интеграция через BasePublisher имеет смысл только когда bamboodom станет "полноценным проектом" для SaaS-подписчиков seo-master. Сейчас это B-side интеграция одного конкретного оператора, а не клиентский проект — избыточно.

### А2. Paste JSON + кнопка «Вставить пример» ✅ с предупреждением

Согласен с решением. Но **предостережение про Telegram**:

- Telegram обрезает сообщения на **4096 символов**. JSON-блок с 4-5 осмысленными `p`-блоками легко превысит лимит.
- Кавычки в JSON Telegram может **auto-форматировать** («умные кавычки»), если у пользователя включён autocorrect. Это сломает парсинг.
- Пользователь может случайно отправить JSON как **Markdown** (если вставил в `) → текст изменится.

**Что рекомендую добавить в FSM:**
1. Строгий парсинг: `json.loads()` → при ошибке показать понятное сообщение "Неверный JSON: <python error>"
2. Если сообщение >4000 символов — предложить загрузить как файл-документ (.json)
3. Нормализация кавычек перед parse: `text.replace(chr(0x201c), '"').replace(chr(0x201d), '"').replace(chr(0x2018), "'").replace(chr(0x2019), "'")` — страховка от умных кавычек iOS/Mac

Это 10 строк кода, но сэкономит час дебага "почему не работает".

### А3. Sandbox-only в 3A ✅

Правильно. В 3A оператор A учится на тестах, в 4A (когда AI будет писать сам) — переход на production `draft=true`. Чекбокс сейчас — лишняя сущность.

## Блок Б. Содержимое

### Б1. Без картинок в 3A ✅

Согласен. E2E-тест не должен зависеть от дополнительного endpoint'а. Картинки в 4A вместе с AI-generation — там их сразу много, смысл налаживать flow.

### Б2. Product-блоки в 3A ✅

Согласен. Ручной ввод артикула — норм для MVP. Серверная валидация через `blocks_dropped` протестирована, работает. В 4A AI-ассистент сможет подбирать артикулы через закешированный `blog_article_codes`.

### Б3. Локальный валидатор forbidden_claims — важное дополнение

Согласен **не делать в 3A**. Но хочу поднять тему заранее, чтобы не упустить в 4A:

**forbidden_claims — это не только регекс.** В knowledge base (`bamboodom_knowledge.md` раздел 5) запреты делятся на три категории:

1. **Буквальные формулировки** — "самый дешёвый", "скидка 70%". Регекс работает.
2. **Семантические запреты** — "писать конкретные цены в тексте", "сравнения с конкурентами по названию". Регекс **не работает** — Claude может написать "всего 2500 рублей за квадрат" и регекс пропустит, потому что буквально "самый дешёвый" там нет.
3. **Запрет на определённые сертификаты без подтверждения** — "пожарный класс Г1", "E1", "гипоаллергенно". Регекс работает, но нужен аккуратный whitelist для легитимных контекстов ("в статье про нормы пожарной безопасности вообще нельзя писать про Г1? — нет, можно, но не утверждать, что наш материал его имеет").

**Моё предложение для 4A:**
- Регекс-валидатор для **категории 1 и 3** (буквальные + сертификаты) — **быстрый фейл**, блокирует submit.
- Семантическая проверка через **запрос к Claude** ("проверь, не содержит ли этот текст нарушений") — **медленный, но глубокий** layer. Делается на стороне A (у вас уже есть Claude API).
- Серверная валидация на моей стороне (в `blog_api.php`) — пока нет. Я добавлю её **в v1.2** как defensive layer: если через все проверки стороны A что-то проскочит, сервер финально отфильтрует и вернёт в `blocks_dropped`. **Не сейчас** — нужны реальные статьи чтобы понять какие кейсы действительно встречаются.

Но в 3A всё это неактуально. Записываю как roadmap-item.

### Б4. Шаблон JSON ✅

Согласен. Немного дополню — в шаблоне имеет смысл использовать артикул **из наиболее стабильной группы**, которая не исчезнет при переорганизации каталога. Рекомендую:
- Не TK029P (может быть заменён в будущем)
- А **F001** (Concrete Board — стабильное название) или **TK001A** (первый артикул серии A)

Плюс предлагаю в шаблоне сделать `draft: false` **явно**:

```json
{
  "title": "Sandbox test ← правьте это",
  "excerpt": "Короткое описание 1-2 предложения ← правьте это",
  "draft": false,
  "blocks": [
    {"type": "h2", "text": "Заголовок раздела"},
    {"type": "p", "text": "Абзац текста. Разрешены inline-теги: <b>жирный</b>, <i>курсив</i>, <a href='/wpc.html'>ссылка</a>."},
    {"type": "product", "article": "TK001A"},
    {"type": "callout", "style": "info", "title": "Полезно знать",
     "text": "Текст выноски."}
  ]
}
```

Почему `draft: false` явно: **в sandbox** смысла в draft нет (ни один публичный blog_list не покажет sandbox-статью, это же изолированный режим), а дефолтный `draft=true` помешает вам сразу увидеть статью по URL — придётся явно передавать `sandbox=1` в `blog_get`. Чтобы в sandbox по ссылке сразу был открытый preview — `draft: false` в шаблоне.

Формально это расходится с серверным default (true), но для sandbox-шаблона имеет смысл.

## Блок В. Flow

### В1. Redis publish_lock TTL=3 ✅

Согласен Redis, не in-memory. Обоснование:
- У вас бот может быть в несколько процессов (Railway с gunicorn workers) — in-memory lock не пересекается между worker'ами
- Redis у вас уже есть (Upstash после 2A)
- TTL=3 точно совпадает с моим rate-limit

**Добавление:** при 429 — показывайте `Retry-After` из заголовка ответа. Я его отдаю. Не придумывайте своё N.

### В2. Idempotency без действий ✅

Согласен. Upsert по slug — это feature, не баг. Если слепить два раза один и тот же payload — получится одна статья (второй запрос `action_type: "updated"`). Идеально для e2e-тестов.

### В3. Result-экран ✅

Согласен. **Одно замечание:** ссылку на sandbox-статью формируйте как:

```
https://bamboodom.ru/article.html?slug=<slug>&sandbox=1
```

**Не** без `&sandbox=1` — иначе откроется production-статья (которой нет или другая) → 404 или не та. Публичный `article.html` по умолчанию смотрит в `data/blog.json`, с `?sandbox=1` — в `data/blog_sandbox.json`.

Проверьте этот URL-формат при рендере. Сам эндпоинт `blog_publish?sandbox=1` в ответе уже возвращает `url: "/article.html?slug=X&sandbox=1"` — просто используйте его как есть, не модифицируйте.
