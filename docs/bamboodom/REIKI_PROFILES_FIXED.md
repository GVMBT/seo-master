## Что конкретно поменяется в API после заливки

### `blog_article_codes`

- `total: 555` (было 359)
- `reiki: ["R001", "R002", ..., "R050"]` — синтетические стабильные коды (рейки на сайте идентифицируются не артикулом, а номером формы/ширины)
- `profiles: ["XHS-T6", "XHS-T8", ..., "XHS-QB0502", ...]` — реальные коды с сайта. 130 в формате `XHS-*` + 16 специальных (некоторые с кириллицей и пробелами, например `XSH-QB1501 Плинтус с подсветкой` — опечатка в исходных данных оператора B, `XSH` вместо `XHS`; в валидаторе фильтровать не нужно, сайт сам рендерит как есть)

### `blog_context`

`materials[]` теперь содержит `articles_count` для всех 4 категорий:
```json
{"slug": "wpc", "articles_count": 282}
{"slug": "flex", "articles_count": 77}
{"slug": "reiki", "articles_count": 50}
{"slug": "profiles", "articles_count": 146}
```

### `blog_article_info`

Работает для всех 4 типов. Особенности:

- `type: "wpc"` — `current_retail` в ₽/м² (как раньше)
- `type: "flex"` — `current_retail` в ₽/м² (как раньше)
- **`type: "reiki"`** — `current_retail: null`. Причина: рейка — это форм-фактор, цена зависит от выбора декора WPC (цвет/толщина/тип наполнителя). Считать «среднюю» цену некорректно. Возвращаем `retail_note` с пояснением:
  > *"Рейка — форм-фактор. Цена зависит от выбора декора WPC. Точная цена на странице /reiki.html"*
- **`type: "profile"`** — `current_retail` в **₽/шт** (не в ₽/м²!). Формула: `priceYuan * yuan_rate + 0.1*del*usd + 0.1*msk`, затем × multiplier=6, округление до 100. Для XHS-T6 на текущих курсах получается ~400 ₽/шт.

## Что это значит для Сессии 2A

**Ничего менять в вашем коде не нужно.** Формат ответов API не изменился, только контент (длиннее массивы, больше материалов с счётчиками). Ваш UI «Контекст сайта» автоматически покажет все 4 строки в секции "Материалы", экран «Артикулы» — четыре ненулевых категории.

Если у вас уже закешированы ответы (TTL 1 час) — они обновятся при следующем force_refresh или по истечении TTL. Сравнение `cache_key` покажет, что данные изменились (для опциональной логики «Данные обновились» из пункта А3 моего предыдущего ответа).

## Что это значит для Сессии 3A

Claude-драфтер может использовать product-блоки с рейками и профилями. Рекомендации по промпту:

- **Рейки (R001-R050)** — если статья про интерьерный дизайн, зонирование, акустику, акценты — подходящая тема. В тексте называть «реечная панель», не «артикул R001».
- **Профили (XHS-*)** — упоминать в статьях про монтаж («внутренний угол XHS-QB0502», «Т-профиль XHS-T6»). Профили — **вспомогательный** материал, редко главная тема статьи.

Общая рекомендация: product-блок с рейкой или профилем **не** должен быть «одиноким» — обычно идёт **в паре с WPC-артикулом** (например: «использовали серию P TK029P в сочетании с рейкой R003 и профилем XHS-QB0502»).

## Валидатор known-list (Сессия 3A)

Небольшое предупреждение на будущее: если будете делать валидатор известного списка endpoints, помните что **в составе кодов `profiles` попадаются** коды **с пробелами и кириллицей** (~10% от списка):

```
XSH-QB1501 Плинтус с подсветкой
XSH-QB1502 Внеш. угол острый
Цинк. внеш. угол
Блок питания 24V 150W
...
```

Это не баг, это как данные лежат в источнике Дизайн-Сервиса. Если ваш Claude-драфтер будет **фильтровать** для product-блоков только «чистые» коды (regex `^[A-Z]+-[A-Z0-9]+$`) — потеряете ~16 из 146 профилей, но в целом это безопасно для читабельности статей.

Оставить как есть (все 146 проходят) — тоже нормально, просто некоторые коды в карточках товаров будут выглядеть длинновато.
