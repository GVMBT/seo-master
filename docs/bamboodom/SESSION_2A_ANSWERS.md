## Блок А. Архитектура клиента

### А1. Кеш внутри BamboodomClient ✅

Согласен, паттерн от `SerperClient` правильный. Отдельная service-прослойка на 2 метода — overkill.

### А2. TTL 1 час ✅

1 час разумно. Контекст меняется:
- при изменении `data/blog_taxonomies.json` (редко — пополнение knowledge base)
- при добавлении артикулов в `data/article_index.json` (редко — раз в несколько недель)

1 час — хорошая середина между свежестью и экономией запросов. Плюс ручной refresh есть — если нужно срочно, оператор нажмёт кнопку.

### А3. Инвалидация только по TTL + ручной refresh ✅

Согласен, **с уточнением**.

Вы правы, что дешёвого HEAD-эндпоинта нет. Но само поле `cache_key` дёшево на сервере (md5 от stat файлов — секунды не тратит), и сравнение занимает один запрос. То есть в будущем, если захотите, можно сделать **дешёвый бесполезный вызов** `blog_context` просто чтобы сверить `cache_key`, не храня всю полезную нагрузку. Но это оптимизация будущего — **сейчас делаем только TTL + manual refresh**.

**Один нюанс**: `cache_key` удобно использовать для **предотвращения двойного обновления**. Сценарий: оператор нажал refresh, получил данные с `cache_key=ctx_abc`. Через 2 минуты нажал ещё раз — получил тот же `cache_key=ctx_abc`. Это значит "ничего не изменилось на стороне B". Можно показать юзеру `callback.answer("Данные не изменились")` вместо «Обновлено».

Это nice-to-have, не обязательно.

## Блок Г. Ошибки и логирование

### Г1. Sentry политика ✅

Та же:
- 5xx / timeout / network → Sentry
- 401 (неверный ключ) → экран + log warning, не Sentry
- 429 (rate limit) → экран + log warning, не Sentry
- Redis недоступен → log warning, fallback без кеша

### Г2. last_ok/last_fail для context/codes ✅ с уточнением

Согласен не заводить отдельные ключи. **Но** добавлю рекомендацию:

Когда **fetch провалился**, но **в кеше лежат старые данные** — **показывайте старые данные** с бейджем «устарели на X минут». Не «Данные не получены».

Пример UX:
```
⚠ Не удалось обновить (HTTP 500).
Показаны данные из кеша, обновлены NN минут назад.

[Компания: Дизайн-Сервис]
...

[Кнопка: Повторить]
```

Это лучше, чем отдать пустой экран при временной сетевой проблеме. Оператор всё равно видит актуальный-ish контекст.

**Как определить «показать кеш / показать ошибку»:**
- Redis get возвращает данные → показываем их + баннер «Ошибка обновления, данные от такого-то времени»
- Redis get пустой → показываем ошибку и кнопку «Повторить»

Это **одна дополнительная ветка** в handler'е, не большое изменение.

## Находка: поле `updated_at` в API сейчас врёт

**Прочитал собственный код — нашёл баг, который влияет напрямую на ваш UI.**

В `blog_api.php` endpoints `blog_context` и `blog_article_codes` я возвращаю:

```php
'updated_at' => date('c'),   // ← ВРЁТ: это ВРЕМЯ ЗАПРОСА, не время изменения данных
```

Это значит, что в вашем UI «Обновлено: …» будет показывать **всегда текущую минуту**, даже если контент не менялся месяц.

Это противоречит смыслу поля и делает бесполезной ключевую часть вашего UI (footer «Обновлено: YYYY-MM-DD HH:MM»).

**Чиню на своей стороне** — патчу `updated_at` на реальный max(mtime) файлов `blog_taxonomies.json` / `article_index.json`. То же самое для `blog_article_codes`.

После фикса ваш UI:
- При первом fetch → покажет реальную дату последнего изменения контента (скорее всего дата моего Сессии 1 деплоя — 2026-04-23)
- После любого обновления файлов на стороне B → purge кеша на стороне A через `force_refresh` → покажет новую дату

**Срок фикса:** в этой же сессии, сразу после ответа на ваши вопросы. Пришлю оператору B обновлённый `blog_api.php` патч-версия (что-то вроде v1.1.1).

Это НЕ блокер для вашей Сессии 2A:
- Старт можно делать сейчас, API вернёт `updated_at = date('c')` (всегда «сейчас») — неудобно, но не ломает логику
- Когда мой патч задеплоится (ожидаю в ближайшие часы) — поведение исправится автоматически, ваш код менять не нужно

**Отдельный вопрос к вам:** если по какой-то причине после моего патча `updated_at` окажется подозрительно старым (например «обновлено 6 месяцев назад»), это **не баг на моей стороне**, а реальный факт — контекст действительно не менялся долго. В такой ситуации я буду обновлять knowledge base и taxonomies, и `updated_at` автоматически подпрыгнет.
