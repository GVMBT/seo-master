## Блок В. Git/деплой

### В1. Push в `main` напрямую — **ДА**

В проекте нет CODEOWNERS / required reviews (судя по тому что ты не упомянул их в блокировках). Оператор A работает один, PR-review не от кого получать. Рабочий процесс: коммит в main → Railway автодеплой → оператор проверяет в Telegram.

Единственное: **не** делай force-push, **не** переписывай историю. Обычный `git push origin main`.

### В2. Unit-тест на `key_test()` — **НЕТ, в Сессии 1 пропускаем**

Обоснование: `tests/` в проекте не критичны для CI (иначе оператор бы упомянул), реальная проверка идёт через smoke-test из админки в Telegram — это и есть интеграционный тест. Unit-тест с моком httpx добавит файл и время, но не найдёт ошибок, которые не нашёл бы ручной smoke-test.

Когда в Сессии 3 появится AI-валидатор с regex на цены — там тесты полезны. Здесь — нет.

### В3. Commit message — `feat(bamboodom): skeleton integration (admin entry + smoke test)`

Подтверждаю. Или чуть лаконичнее: `feat(bamboodom): admin skeleton + smoke test`. Выбирай на свой вкус в рамках Conventional Commits.

### В4. Линтеры перед коммитом

Прогони:
1. `python -m compileall .` (обязательно — твой план)
2. `ruff check .` (если в `pyproject.toml` настроен ruff — а раз файл есть, вероятно настроен)
3. `ruff format --check .` (если применяется формат)
4. `mypy` — **НЕ прогоняй** без явного запроса: в больших проектах на aiogram часто куча legacy-нарушений, твой PR завалится на чужом коде

Если `ruff check` выдаёт ошибки только в твоих новых файлах — исправляй. Если в чужих — не трогай, это legacy.

`coderabbit.yaml`, `semgrep.yml`, `osv-scanner.toml` — это CI-инструменты, они прогонятся на пуше. Локально не надо.

## Блок Г. Прочее

### Г1. `BAMBOODOM_ENABLED=0` → кнопка не рендерится — **ДА (вариант a)**

Условная вставка в `admin_panel_kb()`:

```python
if settings.bamboodom_enabled:
    builder.row(InlineKeyboardButton(text="🌿 Bamboodom.ru", callback_data="bamboodom:entry"))
```

При `enabled=1` + пустом ключе — кнопка видна, smoke-test показывает ошибку «ключ не настроен». Это ожидаемое поведение для диагностики.

### Г2. Sentry для 5xx/таймаутов, НЕ для 401/429 — **ДА, согласен**

Правильное разделение:
- 401 — пользовательская ошибка конфигурации (ключ неверный или отсутствует) → лог + экран
- 429 — rate limit, ожидаемое поведение → лог + экран с `Retry-After`
- 5xx / ConnectTimeout / ReadTimeout — проблема на стороне bamboodom.ru или сети → Sentry + экран

### Г3. `admin:panel` как callback возврата — **ДА**

Если проверил по коду — значит так и есть. Следуй реальности.
