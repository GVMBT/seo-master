## 🔴 Критичная системная проблема — галлюцинации цифр

AI понял: для 1500+ слов нужна **конкретика**. И стал добавлять **выдуманные числа**.

### Примеры из всех трёх статей

**Статья 1 (коридор):**
- «грунтовка увеличивает адгезию клея на **30-40%**» ← Категория 5
- «расход клея... **300-350 г/м²**» ← не в KB
- «Разница в весе... **около 2 кг**» ← не в KB
- «на **15-20% больше** клеевого состава» ← не в KB
- «грунтовки — это **150-200 рублей** на квадратный метр» ← Категория 6 (цена в тексте!) + не в KB
- «температурное расширение **0,1-0,15 мм на метр**» ← не в KB

**Статья 2 (спальня):**
- «Цена... **Выше на 15-20%**» ← Категория 6 + не в KB
- «перепады до **5 мм**» ← не в KB (в реальности перепады другие)

**Статья 3 (Q&A flex):**
- «перепады температур **от -50°C до +80°C**» ← не в KB
- «радиус изгиба от **5-7 см** ... **30-50 см**» ← не в KB
- «расход клея **3-5 кг/м²**» ← не в KB
- «**Срок службы... 25 лет**» ← **Категория 5 critical** (юридически опасное обещание)
- «стойкость цвета подтверждена **испытаниями на UV-старение**» ← Категория 3 (несуществующие испытания)
- «**30+ лет** в интерьере» ← Категория 5
- «**наждачной бумагой зернистостью 120-180**» ← не в KB

### Почему это опасно

1. **Юридически:** «25 лет на фасаде» превращается в оферту. Клиент может через 20 лет потребовать компенсацию.
2. **Для репутации:** если у Дизайн-Сервиса нет испытаний на UV-старение, а в статье написано «подтверждено испытаниями» — это ложь, которую легко проверить.
3. **Для SEO:** Google/Яндекс через LLM детектируют несостыковки между статьями одного сайта. Если в статье 1 «расход клея 300-350 г», а в статье 5 «180-220 г» — это сигнал «сайт пишет наугад».

### Решение: RULE 4 для промпта v6

```
RULE 4 — ЦИФРЫ, ПРОЦЕНТЫ, СРОКИ, ТЕМПЕРАТУРЫ:

Конкретные числовые значения допустимы ТОЛЬКО если они есть в knowledge base.

ЗАПРЕЩЕНО придумывать:
- Расход материалов ("300-350 г/м²", "3-5 кг/м²")
- Сроки службы ("25 лет", "30+ лет", "прослужит N лет")
- Проценты улучшения ("на 30-40% лучше", "на 15-20% больше")
- Температурные диапазоны ("от -50°C до +80°C") — только если в KB
- Цены в рублях ("150-200 рублей за м²") — цены ТОЛЬКО через product-блок
- Размеры/радиусы/зернистость без источника в KB
- "Испытания", "сертификаты", "тесты" — если их нет у Дизайн-Сервис

Если факт требует числа и числа нет в KB — переформулируй общими словами:

✗ "расход клея 300-350 г/м²"
✓ "расход клея зависит от толщины панели и ровности основания — уточните у менеджера"

✗ "срок службы 25 лет"  
✓ "материал долговечен при правильном монтаже"

✗ "выдерживает -50°C до +80°C"
✓ "подходит для всех климатических зон России"

✗ "подтверждено испытаниями на UV-старение"
✓ "декоративный слой устойчив к выцветанию"

✗ "грунтовка 150-200 руб/м²"
✓ [упоминать цены в тексте запрещено]

Если факт критичен и числа нет в KB — ЛУЧШЕ НЕ УПОМИНАТЬ факт вообще,
чем придумать число.
```

## 🔴 Валидатор пропускает галлюцинации артикулов в тексте

**Критичная дыра в pre-publish валидаторе.**

В Статье 1 и Статье 2 упоминается `TK042A`:
- В Статье 2: через product-блок → **валидатор поймал**
- В Статье 1: только в тексте p-блока → **валидатор не заметил**

Цитата из Статьи 1:
> «На проекте коридора 12 м² в Сочи **артикул TK042A** из серии A с текстурой камня...»

`TK042A` не существует. Серия A: `TK001A-012A, 110A-112A, 157A-159A, 187A-195A`. Номер 042 только в серии P.

### Решение: расширить валидатор на text-блоки

Regex для извлечения артикул-кодов из p/h2/h3 блоков:

```python
import re

ARTICLE_CODE_PATTERN = re.compile(
    r'\b(TK\d{3}[A-Z]{1,3}|F\d{3}|R\d{3}|XHS-[A-Z0-9]+|BJL?\d{2}|TK\d{3}H)\b'
)

async def validate_article_codes_in_text(blocks: list[dict], valid_codes: set[str]) -> list[str]:
    violations = []
    for i, block in enumerate(blocks):
        text = block.get('text', '') or ''
        if block.get('type') == 'list':
            text += ' ' + ' '.join(block.get('items', []))
        for match in ARTICLE_CODE_PATTERN.finditer(text):
            code = match.group(0)
            if code not in valid_codes:
                violations.append(
                    f"Unknown article code '{code}' mentioned in block #{i} text"
                )
    return violations
```

Вызывать **дополнительно** к существующей проверке product-блоков. Результаты включать в тот же список violations которые оператор A видит в preview.

Это поймает **все упоминания артикулов в тексте**, не только в product-блоках. Важно для случаев типа «панели серии P применяются в...» или «артикул TK042A в коридоре...».
