## 🚨 КРИТИЧЕСКОЕ: тихий запуск (дополнение к Д3)

Вы предлагаете: **production `draft=true` по умолчанию + скрытая sandbox**.

**Я возражаю.** Ставлю обратное: **sandbox по умолчанию + ручное переключение оператором на production после 5-10 успешных генераций**.

**Обоснование.**

В 3A вы опубликовали одну статью — красиво отработало. В 4A AI будет писать самостоятельно. Реальный риск для первых генераций:

1. **Галлюцинация фактов про компанию** — AI может выдать «работаем с 1995 года» вместо 1997, «прямая поставка из Турции» вместо Китая, или придумать цифры вроде «более 500 довольных клиентов в год». Валидатор regex такое не поймает.
2. **Некорректные product-блоки** — AI поставит `TK029P` там, где нужен `TK029B`, или использует артикул который реально существует, но в статье про ванную порекомендует серию для сухих помещений. Сервер валидирует только сам код, не семантику.
3. **Мягкие нарушения тона** — AI может написать «гипоаллергенный» (без сертификата — запрещено), но перефразированно — «безопасен для астматиков и аллергиков». Layer 1/2 regex такое не ловит.

Эти кейсы — **именно то, что ловится человеческим глазом за 2 минуты**, но сервер их пропустит. И если сразу production — оператор B увидит их **после** публикации (в кабинете модерации). Это нормально если таких статей 1-2. Но если AI зальёт 10 статей подряд с однотипной галлюцинацией — это уже не «забавный факт модерации», это уборочная работа на полдня.

**Что я хочу:**

1. **Первые 5-10 статей** → `?sandbox=1` флаг. Они попадают в песочницу, не в production. Я их читаю **целиком** через мою админку (вкладка «🧪 Песочница»), фидбэчу вам какие паттерны улучшить.
2. **Когда я подтвержу, что качество стабильно** — вы убираете `&sandbox=1` из своего клиента, статьи начинают идти в production `draft=true`, дальше модерация в кабинете по одной.

**Как это делать в UI:**

На preview-экране FSM — радио-группа:
```
□ Production draft=true (после апрувала стороной B)
☑ Sandbox (тест)
```

По умолчанию sandbox. Production появляется только если сторона A **явно переключает** (например, через скрытую команду в админке бота — не обычную клавиатуру). Тогда оператор A не перепутает.

**Это блокер для запуска 4A со стороны B.** Если не согласны — обсудим. Но пока так.

## 🚨 КРИТИЧЕСКОЕ: страховка на сервере

Для перехода на production я добавлю **два средства защиты на моей стороне**. Это не затрагивает вашу работу, я делаю отдельно, между 4A и первой production-генерацией:

1. **Rate-limit на черновики**: не более 5 новых (created) черновиков в сутки от одного X-Blog-Key. Отдельно от существующего rate-limit (1 publish/3s — он против spam, а мой — против «утечки» в случае багов).
2. **Лог публикаций**: каждый `blog_publish` пишется в `data/blog_publish_log.json` — `{timestamp, slug, action_type, blocks_dropped_count, sandbox, source_ip}`. Нужно для аудита постфактум («а что произошло 23 апреля с 14:00 до 15:00?»).

Эти изменения — **моя работа на следующей сессии** (v1.1.4), не блокируют 4A. Просто сообщаю, что это будет.

## Блок Б. Формат AI-вывода

### Б1. Structured JSON + strict parse (a) ✅

Согласен. Tool use (c) элегантнее, но в production — опыт показывает что Anthropic tool-use иногда галлюцинирует имена инструментов или пропускает обязательные аргументы. JSON parse + retry — надёжнее.

**Один совет:** если JSON невалиден **ровно 1 раз** — retry со **строгим промптом** («Твой предыдущий ответ не был валидным JSON. Верни ТОЛЬКО массив блоков, без markdown, без объяснений, без обёрток ```json ```»). Если второй раз — `AIGenerationError`, показ оператору.

Retry до 2 раз не надо, это дорого. Один retry — оптимально.

### Д2. FSM flow ✅ с упрощением

Согласен с потоком. **Упростите `choose_articles`:**

Пусть **AI сам выбирает** артикулы из `blog_article_codes` + `blog_article_info` по ключевому слову. Оператор в FSM артикулы вручную не выбирает.

Причина: это **вся суть AI-пайплайна** — оператор ввёл «WPC для ванной», AI подобрал 2-3 артикула из подходящих (например TK103B fabric + TK170B stone для мокрой зоны). Ручной выбор убивает автоматизацию.

Итого FSM:
```
choose_material → enter_keyword → generating →
preview → validating → result
```

5 шагов, не 6. Оператор может только согласиться / отменить / перегенерировать с другим keyword.
