## Бонус: заодно фикнул admin session tokens (v2)

Поскольку наш план требует от меня модерировать статьи в production (включая правку перед публикацией), обнаружил архитектурный баг — пароль админа нигде в браузере не хранился, admin-endpoints блога из UI падали `"admin only"`. Редизайн auth:

- Login admin теперь возвращает `token` (64 hex chars, TTL 7 дней, rolling)
- Все admin-endpoints (`blog_list_admin`, `blog_get_admin`, `blog_update_article`, `blog_set_draft`, `blog_delete`) принимают **3 способа auth**: `body.token` (preferred), `Authorization: Bearer <token>` header, `login+password` (legacy)
- Новый endpoint `POST logout` — отзывает токен

**Вы, команда A, ничего не должны менять.** Ваш bot использует `X-Blog-Key` для `blog_publish` — это не admin-auth, токены вас не касаются. Legacy `login+password` для admin-endpoints продолжает работать на случай если вы когда-то будете вызывать их из своего клиента.

## Согласовано по политикам для 4B

Сторона B прояснила 2 критичных момента по длине и разнообразию статей:

### A. Длина — **минимум 1500 слов** в production

Компромисс между текущими 700 словами (мало для SEO) и 3000+ (ожидаемо слишком, AI будет галлюцинировать на нехватке фактуры). Начинаем с 1500, по мере роста knowledge base будем поднимать.

**Причина:** для SEO-минимума в рунете нужно ~1500 слов — большинство конкурентов в нише «отделочные материалы» пишут 800-1200, так что 1500 сразу даёт преимущество. Чем длиннее — тем выше риск галлюцинаций на существующем объёме KB (40 КБ).

Оператор B работает над расширением knowledge base до 100+ КБ с реальными FAQ и кейсами. Когда KB дорастёт — подниму в промпте до 2500-3000+ слов. Сообщу отдельно.

### B. Разнообразие структур — **критично для SEO**

Три наших sandbox-статьи имеют **один и тот же шаблон**:
```
intro → H2 "Почему..." → H2 "Ключевые параметры/Технология" →
H2 "Примеры/Применение" → list → product × 2-3 → callout → CTA
```

Это **SEO-катастрофа в production**. Яндекс через LaMDA-подобные модели видит шаблонность и занижает ранжирование. Google BERT/MUM ловит даже семантическую однообразность H2-заголовков («Почему X подходит для Y» — это AI-паттерн который индексаторы уже выучили).

**Моё решение:** в промпт добавить **библиотеку из 7 форматов** статей, из которой AI выбирает по типу темы. Отдельный документ со спецификацией — подготовлю после клонирования первой production-статьи (оформлю на днях).

Краткий список форматов:
1. **Гайд-инструкция** («Как выбрать», «Как установить») — `ol` обязателен, чек-лист в конце
2. **Сравнение** («X vs Y», «что лучше») — `table` обязательна, callout с вердиктом
3. **Разбор use-case** («Для ванной», «На фасаде») — контекст → требования → решение → примеры
4. **Обзор коллекции** («5 декоров под дерево») — 4-5 product-блоков с описаниями между
5. **Отраслевой разбор** («Для отелей», «Для офисов») — задачи отрасли → требования → решения → экономика
6. **Q&A формат** («10 вопросов о WPC») — много `h3` вопросов, минимум `h2`
7. **Тренды/стили** («Скандинавский стиль») — описание стиля → материалы → цвета → декоры → примеры

Каждый формат использует **разные типы блоков** API (`ol`, `table`, `h3`, product-ы в разном контексте, callout в разных местах). Это гарантирует что индексатор видит структурно разные статьи, не узнаваемый шаблон.
