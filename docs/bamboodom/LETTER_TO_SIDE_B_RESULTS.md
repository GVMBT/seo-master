### Новый блок RULE 4.1 — запрет выдуманных конкретных объектов

По вашему отдельному требованию из метрики «0 фейковых кейсов в партии из 10». В промпт добавлен раздел с примерами:

- ❌ «Отель в Сочи на 32 номера», «бизнес-центр в Москве 12000 м²», «сеть ‘Ритм-холдинг’», «архитектор И.И.»

Принцип: «если читатель может погуглить упомянутый объект и не найти — это галлюцинация».

### Hotfix регекса frost-grade

Наш `_ARTICLE_CODE_RE = r"...|F\d{3}|..."` ловил марки морозостойкости (F200, F300) как «unknown article code». Добавлен метод `_is_frost_grade(text, start, code)` — если перед F\d{3} в окне 40 символов есть слово `морозостойкость*` / `класс*` / `марк*` / `grade` — пропускаем.

### Hotfix slipped-codes prefix-aliases

`collect_codes_for_validator(payload)` теперь для каждого `code` с пробелом дополнительно кладёт в allowlist префикс до первого пробела. То есть `XHS-QBDD901` (голый префикс) проходит, как и `XHS-QBDD901 Универс. подсветка` (полная форма). Это под ваш фикс `_splitProfModel` в article.html.

## Баги / открытые проблемы

### Баг (на вашей стороне): фото профилей XHS — все 404

URL партии №4 (profile guide). На странице визуально видно: 3 product-карточки имеют **серый плейсхолдер** вместо фото. В DOM:

```
src: "https://bamboodom.ru/img/prof/xhs-l10.jpg"  → loaded:false, natural:0x0
src: "https://bamboodom.ru/img/prof/xhs-u11.jpg"  → loaded:false, natural:0x0
src: "https://bamboodom.ru/img/prof/xhs-t8.jpg"   → loaded:false, natural:0x0
```

Все три → 404. Корневая причина:
- В `blog_article_index_full` для всех profile-артикулов `cover_img: ""` (пустое поле).
- Сервер рендера строит путь по convention `/img/prof/{lowercase-code}.jpg`, но физических файлов фото для XHS-профилей в `/img/prof/` **не выложено**.

Сравните с другими категориями (всё работает):
- wpc: `cover_img: "/img/Plenki/TK029P.jpg"` — файл есть
- flex: `cover_img: "/img/flex/Concrete_Board.webp"` — файл есть
- reiki: `cover_img: "/img/reiki/reiki_28.webp"` — файл есть
- profiles: `cover_img: ""` — **пустое + файлов нет** ← баг

**Что нужно от вас:** либо залить фото профилей в `/img/prof/` (146 артикулов, по одному фото на тип = ~30 уникальных фото с учётом группировки `category_id`), либо заполнить `cover_img` в API для тех профилей, у которых фото где-то уже есть.

Это не блокер для production — текст и описание product-блока работают, ссылки тоже, просто карточка идёт без фото. Но для коммерческого блога это визуально слабо.

### Замечания по содержанию

- Статья №2 (сканди) использует только серию P (5 артикулов TK029P/030P/031P/119P/156P). Это правильный выбор для темы (P = wood, светлые), но возможно стоит дать AI намёк что для скандинавского стиля можно расширить выборку — например светлые fabric (P-fabric texture) или тонкие XHS-T6/T8 в светлых цветах. v13 backlog.
- Статья №3 (reiki) использует пары reiki+wpc артикулов — что соответствует нашей политике в промпте «product reiki всегда в паре с wpc». Это хорошо.
- Статья №4 (profiles) — правильно использует базовые типы (L, U, T) как образцы. Не упоминает специализированные QBDD901 / NQ50 / другие сложные профили. Это нормально для guide-формата.

После v13 — 2 темы:
