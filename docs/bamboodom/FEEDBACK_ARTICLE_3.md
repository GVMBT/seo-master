## 🔴 Единственный блокер — «серия Wooden»

**В статье:**
> *«WPC-панель TK029P из **серии Wooden**»*
> *«WPC-панель TK042P из **серии Wooden**»*

**Реальность:**
- Серии WPC: P, A, B, C, D, M, S, Y, G, BJL, BJ (буквенные коды, 11 серий).
- `TK029P`, `TK042P` — оба принадлежат **серии P**.
- «wood» — это **тип текстуры** (texture_type), не название серии.

**Причина галлюцинации:** скорее всего AI получил из `blog_article_info` поле `"texture_type":"wood"` и превратил его в «серия Wooden» по инстинкту (как делают маркетологи в западных каталогах — «Wooden Line», «Classic Wooden»).

**Правка в промпт (одна строка):**
> *«Серии WPC обозначаются ОДНОЙ БУКВОЙ: P, A, B, C, D, M, S, Y, G, BJL, BJ. Последняя буква в артикуле (пример: TK029**P**) — это серия. Поле `texture_type` из blog_article_info (wood/stone/fabric/leather/marble/metallic/pattern/art) — это ХАРАКТЕРИСТИКА ПОВЕРХНОСТИ артикула, НЕ название серии. Никогда не пиши "серия Wooden", "серия Stone". Правильно: "серия P", "серия P с декором wood", "артикул из серии A с текстурой камня".»*

## 🟡 Мелочи (не блокеры)

### Блок `list` вместо `table`

В разделе «Монтаж» использован маркированный список с двумя пунктами:
```json
{"items":[
  "WPC — клей для WPC, ровная стена, алюминиевые профили в стыках",
  "Гибкая керамика — плиточный клей для наружных работ..."
]}
```

Для сравнительной статьи **лучше смотрелась бы таблица**. API поддерживает `type: "table"`.

Мелочь на будущее — промпт: *«Для сравнительных статей, где сопоставляются 2+ материалов, используй блок type=table (header + rows), не маркированный список.»*

### Нет product-блока от флекса

В статье WPC-артикулы (TK029P, TK042P), но артикул флекса (F001 или F030 для колонны) не упомянут, хотя по смыслу статьи было бы логично.

Мелочь — на будущее добавить в промпт: *«В сравнительной статье обязательны минимум 2 product-блока: по одному от каждого сравниваемого материала».*
