**Mеханизм добавления новых товаров в sitemap до сегодняшнего дня:** никакой. Файл собирался вручную в прошлой итерации, после этого новые товары в админке сайта не триггерили его пересборку. Это и был пробел, который вы вскрыли.

**Что точно не попало в старый sitemap:** 38 новых flex-текстур (Wood Concrete Board, Cliff Stone, Rust Series, Gilded Series и др.), 10 новых WPC-текстур (TK181D, TK182B–186H, TK192A–195A, TK199B–201B), и любые обновления цен/спеков (lastmod не апдейтился).

### Q4 — TG-отчёт за последние дни

Точные цифры у Александра в Telegram. Из логики кода (см. ниже Q5) ясно что за последнюю неделю cron должен показывать в TG 0 (ноль) `submitted` каждый день — это и был симптом. После сегодняшнего деплоя ситуация изменится: при следующем запуске cron подхватит свежий sitemap и отправит дельту.

### Q5 — логика дедупликации в cron

Изучил `auto_reindex_cron.php` (он же `auto_reindex_v2.php`):

```php
$candidates = array_diff(
    $sitemapUrls,                  // что есть в sitemap.xml сейчас
    array_keys($alreadySubmitted), // что уже было в data/yandex_reindex_log.json
    array_keys($yandexQueue)       // что уже в очереди Яндекса
);
$toSend = array_slice($candidates, 0, $BATCH_SIZE);  // BATCH_SIZE = 150
```

Это значит:
- **150 URL/сутки** — это **верхний лимит**, фактическая отправка = `min(150, count($candidates))`.
- Cron шлёт каждый URL **ровно один раз**. Если URL уже в snapshot — он не отправится повторно.
- После того как 861 URL (статичный sitemap) был распределён по дням за первые ~6 дней (861 / 150 ≈ 6 дней), последующие запуски стали отправлять 0/сутки.
- При появлении нового URL в sitemap.xml — он попадёт в `candidates` и уйдёт в очередь следующим запуском cron.

### Q6 — lastmod не учитывается

**Подтверждаю.** Cron смотрит только на наличие URL в snapshot, lastmod игнорирует. Это значит:
- Обновления цен/описаний на существующих страницах не доходят до Я.Вебмастера через наш cron.
- Если страница изменилась, но URL остался прежним — Яндекс узнает только когда сам найдёт изменение через регулярный обход.

Это второй пробел, который пока **не закрыт**. Зафиксировано в backlog (см. ниже).

### Триггер автоматической регенерации

В админке сайта (cabinet.html) добавлю кнопку «🗺 Регенерировать sitemap.xml» рядом с уже существующей для блога — на следующей итерации (вторник). Сейчас регенерация:
- Ручная — через POST endpoint (ваш бот может дёргать по необходимости).
- Автоматическая на одно событие — при production-публикации статьи блога (это уже делает `blog_regenerate_sitemap`, обновляет `sitemap_blog.xml`).

Авто-регенерация при изменении товаров (вариант 2 из вашего письма) — отложена. Это требует точечных триггеров в нескольких местах админки (где правится товарный JSON), и пока что мы не уверены что эти места стабильны. Раз в сутки регенерации через cron достаточно.

## По вашему предложению о кнопке в боте

**Да, добавьте** в админку бота кнопку «🗺 Регенерировать sitemap.xml (общий)». Она вызывает наш новый `POST blog_regenerate_sitemap_full`. Не дублирует с нашей админкой потому что наша делается завтра и оператор будет работать в одной из них.

Кнопка для блога (`POST blog_regenerate_sitemap`) у вас уже есть — оставьте отдельной, эти два endpoint'а независимы:
- `blog_regenerate_sitemap` → перестраивает `sitemap_blog.xml` (только статьи блога)
- `blog_regenerate_sitemap_full` → перестраивает `sitemap.xml` (весь сайт + текстуры/профили/рейки)

## Замечание про lastmod

Текущий генератор пишет `<lastmod>` = текущая дата для всех URL'ов. Это технически некорректно (URL мог не меняться годами), но для нашего случая — это **сигнал Яндексу что pages обновились** и стоит их перепроверить. После того как все 574 URL'а пройдут через recrawl queue (~4 дня по 150/сутки), мы можем переключить логику на «реальный mtime файла на сервере» — тогда lastmod снова станет точным маркером изменений.
